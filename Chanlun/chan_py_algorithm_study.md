## 1. 结论先行

这套 Python 实现给我们的最重要启示是：

**“当前级别顶底”不应通过均线、MACD 或固定 ATR 对所有局部分型直接分类，而应当作为结构递归生成的结果。**

它的主链路是：

```text
原始 K 线 KLU
  -> 去包含后的合成 K 线 KLC
  -> 局部三 K 分型 FX
  -> 满足跨度、分型区间、端点极值要求的笔 BI
  -> 由笔的特征序列确认线段 SEG
  -> 在线段内部生成中枢 ZS
  -> 比较进出中枢力度，识别背驰与买卖点 BSP
```

因此需要区分三个完全不同的问题：

1. **这里是否存在局部分型？** 由合成 K 线三元素关系回答。
2. **该分型是否有资格成为一笔端点？** 由笔的结构约束回答。
3. **该端点是否代表更高一层的转折？** 由线段、特征序列、中枢和次级别背驰回答。

当前 Pine Phase 1C 将“反向跨度 + ATR 幅度”直接解释为“当前级别分型”，混合了第 2、3 个问题。它能减少标签，但不是严格的级别生成算法。建议将 ATR 降级为：

- 调试统计；
- 候选强度评分；
- 极端波动保护；
- 交易风险过滤；

而不是决定一个分型是否属于缠论当前级别的硬门槛。

核心增量调用顺序为：

1. `CChan.load_iterator()` 按时间顺序加入每根原始 K 线，并建立父级/次级 K 线对应关系。
2. `CKLine_List.add_single_klu()` 先计算 MACD 等指标，再尝试把原始 K 线加入最后一根合成 K 线。
3. 若不能包含合并，则封口旧 KLC、创建新 KLC，并用最后三根 KLC 更新中间 KLC 的分型。
4. 新确认的分型交给 `CBiList.update_bi()` 更新笔。
5. 逐步模式下，只要笔或虚笔发生变化，就重算不确定尾部的线段、中枢和买卖点。
6. 批处理模式在数据全部读完后统一补虚笔并计算线段、中枢和买卖点。

这是一条事件驱动的分层状态机，而不是一次对历史极值做窗口扫描。

## 3. 第一层：原始 K 线与包含处理

### 3.1 数据对象分层

项目将原始 K 线称为 `CKLine_Unit`（KLU），将包含处理后的 K 线称为 `CKLine`（KLC）。一个 KLC 内保存一至多根 KLU。

这与 Pine 当前的 `raw bar -> merged bar` 设计一致。需要继续坚持两套索引：

- 原始 bar index：用于定位图上的真实极值；
- 合成 K 序号：用于跨度、分型和笔的结构判断。

### 3.2 包含判定

`CKLine_Combiner.test_combine()` 将相邻区间分为四类：

- 前者包含后者；
- 后者包含前者；
- 高低点同时下移；
- 高低点同时上移。

普通 K 线处理中，前两类都进入 `COMBINE`。后两类分别给下一根 KLC 确定 `DOWN` 或 `UP` 方向。

### 3.3 有方向的合并

若当前 KLC 方向为 `UP`：

```text
mergedHigh = max(oldHigh, newHigh)
mergedLow  = max(oldLow,  newLow)
```

若方向为 `DOWN`：

```text
mergedHigh = min(oldHigh, newHigh)
mergedLow  = min(oldLow,  newLow)
```

项目还专门处理一字 K，防止一字 K 把合并结果错误压缩。

### 3.4 对当前 Pine 的含义

当前 Pine 的方向合并主体与此相符，但有一处语义差异值得记录：Python 第一根 KLC 默认以 `UP` 初始化；Pine 在方向未知时通过收盘价/区间中点 bootstrap。后者更显式，但必须用回放测试证明：

- 首段包含链在重新加载后结果一致；
- bootstrap 不会因为图表截取起点不同而长期改变后续结构；
- 同一份 OHLC 在批处理和逐 bar 回放中生成完全相同的 sealed merged 序列。

## 4. 第二层：局部分型只是“原料”

### 4.1 分型只在合成 K 上计算

当新 KLC 出现后，程序才用 `前一根—中间根—后一根` 更新中间 KLC 的分型。换言之，中间 KLC 至少等右侧一根合成 K 才能确认。

普通模式采用严格高低点关系：

```text
顶分型：B.high > A.high、B.high > C.high，
        B.low  > A.low、 B.low  > C.low

底分型：B.high < A.high、B.high < C.high，
        B.low  < A.low、 B.low  < C.low
```

这对应当前 Pine 的 `STRICT` 模式，而不是默认 `LOOSE` 模式。

### 4.2 分型不携带“级别标签”

Python 实现不会在检测分型时问：

- 是否离均线足够远；
- MACD 是否金叉/死叉；
- 反向幅度是否达到一倍 ATR；
- 这是不是日线级别分型。

分型只是合成 K 序列上的局部事实。它能否成为结构端点，要交给下一层笔状态机决定。

这解释了为什么直接显示全部 detected fractal 必然密集：**局部分型密集不是检测错误，而是把底层原料错误地当成最终结构展示。**

## 5. 第三层：笔状态机才是第一道“定级”机制

`CBiList` 是本次最值得移植的部分。它不是把每个顶底分型直接相连，而是维护一个会更新、替换、回滚的尾部状态。

### 5.1 第一笔

第一笔出现前，分型进入 `free_klc_lst`。新分型会向前寻找类型相反、且能够满足成笔规则的历史分型。找到后才建立第一笔。

这比“把遇到的第一个分型直接设为当前级别锚点”更稳健，因为起始窗口可能恰好从趋势中部开始。

### 5.2 同类型分型不新增端点

若新分型与当前笔尾同型：

- 上升笔只接受更高或相等的顶；
- 下降笔只接受更低或相等的底；
- 不够极端的同型分型被忽略。

因此一串连续顶只保留最高顶，一串连续底只保留最低底。它不是事后批量“消重”，而是笔尾的在线极值更新。

### 5.3 反向分型必须通过三组结构门槛

相反类型分型只有同时通过以下规则才会生成下一笔。

#### A. 合成 K 跨度

默认严格笔要求：

```text
endMergedIndex - startMergedIndex >= 4
```

注意这是索引差至少 4，即端点之间有 3 根合成 K、端点计入共 5 根。当前需求文档中的“至少间隔 4 根”需要统一为这个精确定义，避免 off-by-one。

如果配置 `gap_as_kl=true`，价格缺口可以补算为一根 K；当前默认配置却设为 `false`，说明实际默认不会用缺口缩短笔距。

#### B. 两端分型区间关系

`bi_fx_check` 提供四种严格度：`loss / half / strict / totally`。当前默认是 `strict`。

以底到顶的上升笔为例，严格模式不仅要求终点顶高于起点附近高点，也要求终点附近低点与起点三元素低点形成足够分离。它比较的是两个分型三元素的包络，而不是只比较两个极值价格。

这项约束能大量删除震荡中“虽有局部顶底，但两组三 K 区间没有真正拉开”的伪笔。

#### C. 终点必须是区间极值

默认 `bi_end_is_peak=true`。候选终点必须是从起点到终点这一整段内相应方向的最高点或最低点。

例如底到顶时，如果中间已经出现比候选顶更高的 KLC，则候选顶不能成为笔尾。反向同理。

### 5.4 `bi_allow_sub_peak` 的真实含义

配置名容易误读：

- `true`：允许次高/次低点形成后续笔，不执行跨笔回撤式重构；
- `false`：在一组严格条件成立时，可以删除最后一笔，并把更早一笔的终点更新到新的极值。

也就是说，关闭次级极值后，算法有能力处理类似：

```text
T1 -> B1 -> T2 -> B2
```

其中后续结构证明 `B1/T2` 不足以作为独立层级转折时，对尾部做结构性撤销，而不是简单按固定振幅过滤。

这与我们此前提出的“3 月份前顶—假底—假顶—后底合并”方向一致，但 Python 的实现条件更严格：它检查笔方向、前一笔起点、区间极值以及新旧端点关系，并非只做四点价格支配比较。

### 5.5 实笔、虚笔和稳定边界

项目明确区分：

- `is_sure=true`：由已确认反向分型形成的实笔；
- `is_sure=false`：由当前尚未形成标准分型的走势方向推演出的虚笔。

虚笔可以：

- 随新高/新低延长最后一笔；
- 在满足虚拟成笔条件时生成反向尾笔；
- 下一次实分型到来时删除、恢复或重新展开。

一个重要工程事实是：**`sure` 并不等于“从此绝不修改”。** 最后一根实笔仍可能被后续同型更极端分型延长；在禁止次级极值时，尾部甚至可能撤销一笔重构。因此 UI 至少需要四种状态：

1. `DETECTED_FX`：局部分型，仅调试可见；
2. `VIRTUAL_END`：虚笔端点，实时变化；
3. `SURE_TAIL`：已由分型成笔，但仍处在可更新尾部；
4. `STABLE_HISTORY`：已越过回滚边界的历史端点。

如果坚持“confirmed 后永不改”，就不能逐字复现该 Python 算法；需要把 `SURE_TAIL` 画成 tentative，把更早的端点才画成 confirmed。

## 6. 第四层：线段通过特征序列再次提升级别

一笔成立仍不等于它的端点就是更高层转折。线段算法使用特征序列进一步压缩。

### 6.1 特征序列

对于上升线段，程序取其中的下降笔形成特征序列；对于下降线段，取上升笔形成特征序列。特征元素本身再次执行包含合并，然后寻找三个特征元素构成的顶/底分型。

这本质上是同一结构规则在“笔序列”上的再应用：

```text
K 线包含 -> K 线分型
笔的特征序列包含 -> 特征序列分型 -> 线段端点
```

因此“级别”主要来自结构递归，而不是来自绝对波动阈值。

### 6.2 实际突破校验

`CEigenFX.actual_break()` 防止因包含合并而产生“图形上像分型、实际上第三元素没有突破第二元素”的假线段终结。

若特征分型存在缺口，还不能立即确认线段结束；程序会继续寻找反向分型作为附加确认。找不到足够证据时，线段保持不确定。

这对现代高波动行情尤其重要：单根事件 K、隔夜跳空、期权到期附近的尖刺不能只凭局部极值就宣布高级别反转。

### 6.3 不确定尾部重算

每次更新线段前，程序删除末尾不确定线段；若最后一个已确定线段的特征分型第三元素包含不确定笔，也会回退并重算。

但已越过稳定边界的线段不会每次全量重算。这个“稳定前缀 + 可变尾部”模型非常适合 Pine：

- 历史区只保存确认对象；
- 尾部保留一个有限回滚窗口；
- 每次事件只重建尾部，而不是扫描全历史。

## 7. 第五层：中枢不是分型过滤器，而是震荡解释器

### 7.1 段内中枢

默认 `zs_algo=normal`。算法在线段内取与线段方向相反的笔，寻找区间交集。

对参与构造的笔集合：

```text
ZS.low  = max(eachBi.low)
ZS.high = min(eachBi.high)
```

只有 `ZS.high > ZS.low` 才成立；相切不算重叠。默认不生成一笔中枢。

程序随后记录：

- 进入中枢的笔 `bi_in`；
- 中枢内部笔；
- 离开中枢的笔 `bi_out`；
- 中枢区间 high/low；
- 涉及笔的 peak_high/peak_low。

### 7.2 中枢延伸与合并

新笔若仍与中枢区间重叠，可延伸中枢。相邻中枢默认仅在：

- 属于同一线段；
- 两个中枢区间存在重叠；

时合并。

这正是处理 3 月横向密集走势的第二道机制：即使若干小笔合法，它们也首先被解释成同一中枢内部震荡，而不是每个笔端都升级成日线结构反转。

### 7.3 三种中枢算法

项目提供：

- `normal`：确定线段内计算中枢；
- `over_seg`：允许跨段中枢；
- `auto`：确定部分用段内算法，不确定尾部用跨段算法。

对 Pine 第一版，建议先实现 `normal`。`auto` 很适合实盘，但它依赖可靠的线段稳定边界和尾部重算，不能抢跑。

## 8. 第六层：MACD 在哪里真正参与

### 8.1 不参与分型和成笔

MACD 在每根原始 K 线上持续计算，但不会决定：

- 是否发生包含；
- 是否是顶底分型；
- 是否能成笔；
- 是否能成线段。

均线同样只是可选的 metric/绘图特征，不是结构硬条件。

所以 5/10 日均线可以继续显示，用于观察日线短周期节奏，但不应该作为删除分型或强行定级的主规则。

### 8.2 用于“同方向走势力度比较”

笔对象支持多种力度代理：

- 同向 MACD 柱面积；
- 全部 MACD 柱面积；
- MACD 柱峰值；
- MACD 柱极差；
- 价格斜率；
- 涨跌幅；
- 成交量/成交额及均值；
- RSI 极值。

默认笔级一类买卖点使用 MACD 柱峰值；线段级买卖点默认使用斜率。

### 8.3 背驰的结构前提

中枢背驰并不是“两处分型 MACD 更弱”这么简单。默认逻辑先要求：

1. 已存在中枢；
2. 离开笔突破中枢；
3. 找到进入中枢的同方向笔与离开中枢的同方向笔；
4. 比较两者力度；
5. 可选要求离开笔是相关范围的真正极值。

概念表达为：

```text
outMetric <= divergenceRate * inMetric
```

默认配置文件中的 `divergence_rate` 是无穷大，即结构满足时不以强度淘汰；README 示例建议值为 `0.9`。以后移植时必须显式决定，不可把文档示例误当代码默认值。

### 8.4 盘整背驰

若线段末端没有合适的完整中枢离开结构，程序还会比较同一线段内相隔两笔的同向笔，要求后者创新高/新低但力度衰减，生成 `1p` 盘整背驰候选。

这比在每个分型上比较 MACD 更接近我们需要的一周级别日线交易信号。

## 9. 多级别与“小转大”的实际处理

### 9.1 每个时间级别独立建结构

`CChan` 默认装载日线和 60 分钟线，并为每个级别分别创建完整的：

```text
KLC -> FX -> BI -> SEG -> ZS -> BSP
```

父 K 线与子 K 线建立时间归属关系，但低级别结构不会直接混入高级别数组。

这是很重要的隔离原则：**多周期不是把 60 分钟分型与日线分型放在同一个候选池竞争，而是两个独立状态机最后做结构对应。**

### 9.2 区间套确认

项目示例中的区间套逻辑是：当父级别当前 K 线本身已经处在父级别买卖点位置时，再检查归属于该父 K 的次级别是否出现一类买卖点。次级别背驰用于增强父级别交易时点，不用于改写父级别全部分型。

### 9.3 小转大的工程表达

README 对 `bsp2_follow_1` 和 `bsp3_follow_1` 的说明直接提到小转大：一类点可能因背驰度不足而没有生成，但可配置二类/三类点不强制跟随一类点。

这说明成熟实现对“小转大”的处理不是把一个不合规小分型硬塞进日线分型序列，而是：

- 主结构继续按笔、段、中枢规则运行；
- 尾部允许虚笔/虚段表达尚未完成的反转；
- 次级别背驰提供提前证据；
- 即使一类点缺失，也可允许后续二类或三类结构给出交易确认。

因此，5 月中至 6 月的明显日线顶底若被当前 ATR 算法删除，正确修复路径应是：

1. 检查它能否成为合法笔端点；
2. 若能，不应被固定 ATR 删除；
3. 若当时还不能形成确定线段，则保持 `SURE_TAIL` 或虚段；
4. 用 60m/4h 的背驰或一、二、三类点标注“小转大候选”；
5. 后续日线反向笔/线段确认后再升级状态。

## 10. 对两类已知图形问题的具体回答

### 10.1 3 月中枢内的 3/9 与 3/18 假顶底

建议依次应用以下机制，而不是先加更大的 ATR：

1. **严格 KLC 分型**：先淘汰只有 high 或 low 局部极值、但整体高低关系不成立的宽松分型。
2. **严格笔跨度**：要求合成 K 索引差至少 4。
3. **两端分型包络分离**：使用 `strict`，避免两组三 K 高度重叠仍成笔。
4. **笔尾同型极值更新**：同顶留更高、同底留更低。
5. **整笔端点极值**：候选终点不能被中间更极端值否定。
6. **可选禁用次级极值**：允许在严格条件下撤销尾部小笔并更新更早端点。
7. **中枢吸收**：仍然合法的小笔被解释为中枢内部震荡，不升级为线段端点。

所以“前一个顶和后一个底合并掉中间两点”是可能的结果，但应由笔状态机和线段/中枢层推导出来，不能写成只针对四点的孤立删除公式。

### 10.2 5 月中至 6 月被漏掉的日线顶底

固定一倍 ATR 会在高波动趋势末端产生两个相反错误：

- ATR 被事件大 K 抬高，真正反转达不到阈值；
- ATR 尚未抬高时，趋势中的普通回撤又容易过阈值。

结构算法应先回答它是否形成合法笔，不以 ATR 一票否决。若属于小转大：

- 日线端点先进入可变尾部；
- 次级别背驰和中枢离开失败增加反转证据；
- 日线反向笔成立后确认笔端；
- 特征序列线段成立后确认更高层转折。

这允许真实反转“先预警、后确认”，同时避免为了抓小转大而放松全部历史规则。

## 12. Pine 数据结构建议

### 12.1 KLC 层

继续使用现有字段：

```text
mergedId/startBar/endBar/high/low/highBar/lowBar/direction
```

### 12.2 FX 层

```text
fxId/type/sourceMergedId/confirmMergedId/price/rawBar
```

FX 只追加，不承担当前级别判断。若内存需要裁剪，先保证所有笔引用的 FX 不会被删除。

### 12.3 BI 层

```text
biId
beginFxId/endFxId
beginMergedId/endMergedId
beginRawBar/endRawBar
beginPrice/endPrice
direction
state = VIRTUAL | SURE_TAIL | STABLE
usedToBeSure
```

另设尾部状态：

```text
lastEndFxId
lastSureBiId
stablePrefixBiId
```

### 12.4 线段与中枢层

线段必须引用笔的持久 ID，不能引用会因 `array.shift()` 改变的数组下标。中枢同理引用 `beginBiId/endBiId`。

## 13. 必须写入需求文档的工程约束

### 13.1 身份与索引

- 所有结构使用单调递增持久 ID；
- 数组下标不是身份；
- 原始 bar、合成 K、分型、笔、线段各自有独立坐标；
- 裁剪数组前检查下游引用。

### 13.2 事务式尾部更新

一次事件对并行数组的修改必须满足“全成或全不成”。建议将复杂判断先算入局部变量，最后统一 `push/set/pop`，避免执行一半触发 Pine runtime error 后留下数组错位。

### 13.3 稳定前缀与回滚窗口

- 不全量重算历史；
- 只允许重算稳定边界之后的尾部；
- 线段、中枢、买卖点分别记录自己的 `lastSurePosition`；
- 上游尾部变化时，下游先删除受影响对象，再顺序重建。

### 13.4 状态命名

禁止仅用一个 `confirmed` 布尔值覆盖全部语义。至少使用：

```text
DETECTED
VIRTUAL
SURE_TAIL
STABLE
INVALIDATED
```

### 13.5 回放等价性

同一组 OHLC 数据必须满足：

```text
历史一次加载的结果 == 从第一根逐 bar 回放到最后的结果
```

比较对象至少包括：

- sealed KLC 序列；
- FX 序列；
- sure bi 序列；
- stable segment 序列；
- ZS 边界。

### 13.6 不变量

每根 bar 后检查：

- 所有并行数组等长；
- KLC ID、FX ID、BI ID 单调增加；
- 笔方向交替；
- 相邻笔首尾共享同一端点；
- 上升笔 `endPrice > beginPrice`，下降笔反之；
- stable 对象不能引用 virtual 对象；
- 中枢满足 `high > low`；
- stable 历史不得位于回滚窗口内。

## 14. 验收用例

### A. 包含链

- 连续 5 根互相包含；
- 上行包含与下行包含各一组；
- 一字 K、跳空 K；
- 图表从包含链中部开始加载。

验收：合成 K 高低点、极值原始 bar 和方向可重复。

### B. 同型极值

- 连续三个顶，逐个创新高；
- 连续三个底，逐个创新低；
- 同型但不创新极值。

验收：只更新尾部，不生成同向相邻笔。

### C. 最小跨度边界

- KLC 索引差分别为 3、4、5；
- 有缺口与无缺口；
- 严格笔与非严格笔。

验收：明确验证索引差 4 的边界，不依赖自然语言猜测。

### D. 震荡消元

- `T1-B1-T2-B2`，其中中间反向笔不满足包络分离；
- 满足跨度但终点不是区间极值；
- 合法小笔但全部位于同一中枢。

验收：分别在 FX、BI、SEG/ZS 三层说明哪些点被保留，而不是只检查最终标签数量。

### E. 小转大

- 日线尚未形成确定线段，但 60m 已出现背驰；
- 日线一类点因力度不足缺失，随后出现合格二类点；
- 事件大 K 抬高 ATR 后发生真实反转。

验收：先显示候选，后显示结构确认；固定 ATR 不得否决合法笔。

### F. 尾部回滚

- 实笔尾被同型更极端分型延长；
- 虚笔生成后失效；
- 不确定线段第三特征元素变化；
- 尾部变化导致中枢重建。

验收：稳定历史不变，尾部对象和标签无残留。

## 15. 不建议照搬的部分

成熟不等于无需审计。以下实现不能直接机械复制到 Pine：

1. Python 使用对象引用、双向链表、泛型与 deepcopy；Pine 必须改成持久 ID 和有限数组。
2. Python 可在尾部方便地 pop、递归重算；Pine 需要严格限制回滚窗口和循环次数。
3. 项目缺少与我们 NQ 日 K 截图一一对应的自动化测试，参数仍需基于真实样本回放验证。
4. Python 的多级别装载依赖完整子级别数据；TradingView 高周期请求低周期数据存在数组、条数和性能限制。
5. 原实现的算法配置很多，第一版 Pine 不应全部开放，否则无法确定是哪条规则导致结构变化。
6. 该实现代表一套工程解释，不应把所有默认值当成缠论原文的唯一严格定义。

## 16. 最终建议

下一步不要继续给 Phase 1C 叠加均线/MACD/ATR 过滤条件。最合理的工程顺序是：

1. 将本文件中第 13 节约束合并进 `Chanlun_requirement.md`；
2. 把当前 `level fractal selector` 重构为严格 `BiEngine`；
3. 先只画 detected FX、sure tail 和 stable 笔端三层；
4. 用 3 月震荡、5—6 月反转两个样本逐 bar 验收；
5. 笔稳定后再进入特征序列线段；
6. 线段和中枢稳定后，MACD 才进入背驰；
7. 最后用 60m/4h 次级别处理区间套与小转大。

一句话概括：

> 不要问“这个局部分型的 ATR 是否像日线级别”，而要问“它是否依次通过了笔、线段和中枢的结构证明；若尚未通过，当前处于哪一种可撤销状态”。
